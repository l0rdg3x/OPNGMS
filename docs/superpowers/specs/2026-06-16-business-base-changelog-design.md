# Sub-project 4 ‚Äî Business‚ÜíCommunity base mapping from `opnsense/changelog`

**Date:** 2026-06-16
**Status:** design approved (brainstorm) ‚Äî ready for implementation plan
**Relates to:** the version-aware config editor program (sub-projects 1/2/3 done); the (edition, version)
resolver ([[opnsense-multi-version-edition-support]]).

## Problem

OPNGMS serves a **Business** device the **Community catalog of the OPNsense version its Business release
is based on** (Business edition = a stabilized snapshot of a specific Community version; the core API and
config models are shared). The Business‚ÜíCommunity **base-version mapping** is published as
`business-base.json` (`{"map": {business_version: community_base_version}}`) and consumed by
`app/services/catalog_provider.py::resolve_target`.

Today that map is generated by **scraping `docs.opnsense.org/releases/BE_<v>.html`**
(`tools/opnsense_catalog/cli.py::_fetch_business_pages` ‚Üí `publish.parse_business_base`). Two limits:

1. **Per-major only.** The docs site has one page per Business **major** series (`BE_26.4.html`), so the
   map is keyed by major (`26.4 ‚Üí 26.1.6`). A device on `26.4.1` (whose true base is `26.1.9`) floor-resolves
   to the `26.4` entry and is served the **less-accurate** `26.1.6` catalog ‚Äî minor schema drift.
2. **Fragile source.** It depends on the docs site's HTML layout.

The [`opnsense/changelog`](https://github.com/opnsense/changelog) repo has a `business/` tree with **one
plaintext file per Business sub-version** (`business/<major>/<subversion>`, e.g. `business/26.4/26.4.1`),
and **every file's header states** *"This business release is based on the OPNsense X.Y.Z community
version"* ‚Äî the exact phrase the existing parser already matches. It is version-controlled plaintext and
**complete down to every sub-version**.

## Goal

Generate `business-base.json` from the `opnsense/changelog` repo instead of docs.opnsense.org, producing a
**per-sub-version** map so a Business device resolves to the **most accurate** Community base catalog.
Behavior-preserving for Community devices; no real Business box required; out of scope: editable schemas for
Business **proprietary plugins** (their MVC models are not on public GitHub ‚Äî needs a box).

## Key finding ‚Äî the app/resolver needs NO change

`resolve_target` already floor-resolves the Business map keys:

```python
bmap = (business_base or {}).get("map", {})
be = resolve_version(list(bmap), version)   # greatest BE key <= device version (floor)
cv = resolve_version(community, bmap[be])    # floor-resolve that CE base in the manifest
```

`resolve_version` returns the greatest entry `<= target`. So a **denser (per-sub-version) map flows through
the existing floor logic unchanged**: an exact sub-version hit (`26.4.1 ‚Üí 26.1.9`) wins; an unknown new
sub-version floors to the nearest known Business sub-version `<=` it; an empty/garbled file yields no entry
(the device falls back to current behavior). This is backward-compatible with the old per-major map.

**Therefore the change is confined to the catalog generator (the data source) + the publish workflow + tests.**

## Components

### 1. `tools/opnsense_catalog/cli.py` ‚Äî read the changelog instead of scraping docs
- **Remove** `_fetch_business_pages()` (the `docs.opnsense.org` scraper) and the `business-base` subcommand's
  `--html-dir` / `--fetch` (docs) options.
- **Add** `--changelog-dir DIR`: read every file under `DIR/business/<major>/<subversion>` into
  `{subversion: text}`, then call the existing `parse_business_base`. Filtering:
  - keep only sub-versions whose **major** `>= 25` (the catalog floor, matching `--minimum 25` used for CE);
  - skip directory **symlinks** (older Business majors symlink to community ‚Äî all `< 25` anyway) and any
    file whose name is not a release version (e.g. release-candidate `*.r1` are skipped because the parser
    finds no Business "based on" header in them and `parse_business_base` drops non-matching files).
- **Add** `--fetch` (ops convenience): shallow-`git clone --depth 1 https://github.com/opnsense/changelog`
  into a temp dir and use it as `--changelog-dir`. Network/ops path ‚Üí `# pragma: no cover` (as the old
  `_fetch_business_pages` was).
- The subcommand still writes `business-base.json` = `parse_business_base(pages)` (unchanged output shape).

### 2. `tools/opnsense_catalog/publish.py` ‚Äî reuse `parse_business_base`
- `parse_business_base(pages: dict[str, str]) -> {"map": {...}}` is **reused as-is** (same `_BASE_RE`,
  same never-guess rule: a file without the marker is skipped). Update its docstring: `pages` now maps a
  Business **sub-version ‚Üí its `opnsense/changelog` `business/` file text** (not BE_<v>.html).

### 3. `.github/workflows/publish-catalogs.yml` ‚Äî clone the changelog, drop the scrape
- The step that builds `business-base.json` now: `git clone --depth 1 https://github.com/opnsense/changelog`
  then `python -m tools.opnsense_catalog.cli business-base --changelog-dir <clone> --out business-base.json`,
  uploaded to the rolling `catalogs` release exactly as today. Remove the docs.opnsense.org step. The
  Community catalog generation steps are untouched.

### 4. `app/services/catalog_provider.py` ‚Äî **no change** (see Key finding).

## Resolution semantics (fail-safe, unchanged code path)
- **Exact** sub-version hit: `26.4.1 ‚Üí 26.1.9`.
- **Floor**: a Business device on a sub-version not in the map floors to the nearest known Business
  sub-version `<=` it (e.g. a brand-new `26.4.2` not yet in the changelog floors to `26.4.1`).
- **Never guess**: a Business file without the "based on ‚Ä¶" header produces no map entry; if the whole map
  misses, `resolve_target` returns `None` and `get_catalog` degrades exactly as today.

## Testing
- **`parse_business_base`** (pure unit): feed sub-version‚Üítext using **real changelog header lines**
  (`26.4.1` ‚Üí "based on the OPNsense 26.1.9 community version" ‚áí `26.1.9`; `25.10` ‚Üí `25.7.5`), a file with

  exact `{"map": {...}}`.
- **`resolve_target`** (pure unit): with a per-sub-version map ‚Äî exact hit (`26.4.1 ‚Üí 26.1.9`), sub-version

  each then floor-resolving against a small fake manifest. (Confirms no behavior change is needed.)
- The CLI changelog reader + the Action are `# pragma: no cover` (filesystem/network/ops), as today; a thin
  unit test may exercise the directory walk against a tiny on-disk `business/` fixture if cheap.
- Gate: `cd backend && python -m pytest -q` green + `ruff check app/ tools/`.

## Correction (v0.18.1, 2026-06-16) ‚Äî Business hotfix chains + no floor
The first cut (v0.18.0) matched only *"based on the OPNsense X.Y.Z **community** version"*. Real Business
**hotfix** files instead say *"based on the OPNsense X.Y.Z **business** version"* (e.g. `25.4.3 ‚Üí 25.4.2`,
`24.4.3 ‚Üí 24.4.2`) ‚Äî they chain onto the prior Business release ‚Äî so they fell out of the map. Fixes:
- `_BASE_RE` now captures the kind: `‚Ä¶\s+(community|business)`. `parse_business_base` builds a raw
  `{ver: (referenced_version, kind)}` then **resolves each to a Community base transitively**, following
  `business ‚Üí business` links until a `community` anchor (bounded by `_MAX_CHAIN = 32`, which also breaks
  cycles); an unresolvable chain (referenced version absent) or a cycle yields no entry (never guess).
- The CLI reader **no longer floors at major ‚â• 25** ‚Äî it reads every release-version Business file (still
  skipping symlinked majors + non-release filenames) so chains always resolve and the map is **complete**;
  the Community-catalog floor stays enforced downstream by `resolve_target` (a `<25` base floor-resolves to
  `None` against the `‚â•25` manifest, exactly as before). Verified on the real repo: 38 entries, with
  `25.4.3 ‚Üí 25.1.12` and `24.4.3 ‚Üí 24.1.10` now present.
